# 导出背景内嵌与模拟素材严格对齐设计

## 背景

当前项目已经完成以下能力：

- 预览页面可以正确加载固定背景图。
- 单个与批量素材分类上传可以正常工作。
- 成绩单 / 证书与 Offer 标签已支持按素材存在与否条件渲染。
- 预览与导出已开始共享部分布局参数。

但用户当前又明确指出两类问题：

1. 预览页面背景正常，但最终正式导出的图片背景仍呈现灰色异常，说明导出链路与预览链路在背景资源处理上存在分叉或失真。
2. 上传图片后，生成结果的图片区位置与 `assets/模拟素材.jpg` 不匹配，且文字框的尺寸、边距、字体、行高、对齐方式等细节仍未达到设计要求。

用户已确认本轮采用如下原则：

- 正式导出接受将固定背景图内嵌到导出结果中。
- 上传后布局与文字框细节严格以 `assets/模拟素材.jpg` 为唯一对齐基准。

## 目标

本轮目标如下：

- 修复正式 SVG 与 PNG 导出中的灰底问题，使导出背景图与预览背景图完全一致。
- 将正式导出的固定背景资源从“外部路径引用”切换为“导出时内嵌资源”，避免脱离页面环境后背景丢失。
- 重新校正上传后布局，让图片区、标签、标题牌匾、路径区与 `assets/模拟素材.jpg` 的对应位置严格对齐。
- 重新调整标题框与路径框的文字样式细节，确保尺寸、边距、字体、行高、字重与对齐方式贴近参考规范。
- 保持既有分类上传、标签条件隐藏、批量匹配与正式导出流程不回退。

## 非目标

本轮明确不做以下工作：

- 不新增新的模板结构或新的页面模块。
- 不重新设计新的品牌背景图。
- 不修改分类上传的交互规则。
- 不替换当前技术路线，不引入新的模板引擎。
- 不扩展超出本轮需求的多套模板支持。

## 问题根因判断

### 1. 导出灰底问题

根因判断如下：

- 当前预览直接通过页面中的 `<img>` 加载固定背景路径，因此在浏览器页面上下文中能正常显示。
- 正式导出 SVG 仍通过 `<image href="...">` 引用背景路径，而不是把背景资源实际嵌入到导出内容。
- 当导出的 SVG 脱离当前页面上下文单独打开、下载、二次转换为 PNG，或由浏览器执行 SVG 转 PNG 栅格化时，外部背景资源可能无法稳定解析。
- 一旦背景资源解析失败，SVG 中仍存在灰底兜底矩形，因此用户看到的正式结果就会变成灰色异常状态。

### 2. 上传后布局偏差问题

根因判断如下：

- 当前 `imageLayouts`、`textBlocks`、`layoutReference` 仍未完全以 `模拟素材.jpg` 为唯一坐标基准。
- 预览 CSS 与正式 SVG 虽已共享部分参数，但图片区、文字框、标签与路径区之间仍然存在独立硬编码和比例偏差。
- 文字框样式细节仍带有“近似复刻”而非“严格贴齐”特征，例如标题字号、内边距、路径卡文字位置和 chip 尺寸仍未完全收敛。

## 核心方案

### 1. 正式导出改为内嵌背景资源

本轮正式导出统一采用以下规则：

- 预览仍使用固定背景文件路径，不改变当前预览工作模式。
- SVG 导出时，固定背景图先被读取并转换为可内嵌的数据资源。
- 生成 SVG 时，不再依赖外部相对路径引用背景，而是直接嵌入背景资源内容。
- PNG 下载时，不再基于“可能丢失背景路径的 SVG”二次栅格化，而是基于已内嵌背景的 SVG 生成。

这样可以同时消除以下两种故障：

- 导出的 SVG 单独打开时背景丢失。
- PNG 基于导出 SVG 二次渲染时继续灰底。

### 2. 布局严格以模拟素材为唯一基准

本轮布局回调遵循以下原则：

- `assets/模拟素材.jpg` 是上传后布局与文字框样式的唯一结构基准。
- 当前页面现状不再作为主要参考，只作为回归对照。
- `template-config.example.js` 中的 `imageLayouts`、`textBlocks`、`layoutReference` 作为唯一参数源继续保留，但会整体回调到新的参考坐标。
- 预览层与导出层都必须读取这组回调后的参数，不允许再次出现“预览一个位置、正式结果另一个位置”。

## 模块级设计

### 1. 导出背景模块

涉及文件：

- `app.js`
- 如有必要，增加辅助函数放在 `app.js` 内部，避免引入额外复杂度

设计要求：

- 增加固定背景资源读取与内嵌转换逻辑。
- 区分“预览背景来源”和“正式导出背景来源”。
- 预览继续使用原始路径。
- 正式导出调用 `createPosterSvgMarkup()` 时，背景字段应被解析为内嵌数据。
- PNG 下载逻辑要确保基于已内嵌背景的 SVG 转换。

关键约束：

- 不允许因内嵌背景而影响预览速度或破坏现有页面加载。
- 不允许把用户上传图片误当作正式背景。

### 2. 图片区布局模块

涉及文件：

- `template-config.example.js`
- `app.js`
- `styles.css`

设计要求：

- 重新校正 `1-image`、`2-image`、`3-image` 三种布局。
- 主图、辅图位置严格服从 `模拟素材.jpg` 的图片区结构。
- `1-image`
  - 主图恢复到右上主展示位。
- `2-image`
  - 主图保留右上。
  - 第二张图回到右下左侧位置。
- `3-image`
  - 两张辅图回到底部并列结构，服从参考图的节奏和留白。
- 图片标签 `真实成绩单 / 证书` 与 `真实Offer` 的位置也要同步压回参考位置。

关键约束：

- 如果某一类素材未上传，对应标签仍必须隐藏。
- 不允许出现图片区压住标题框、标签错位、图片超出安全区等问题。

### 3. 标题与路径文字框模块

涉及文件：

- `template-config.example.js`
- `styles.css`
- `app.js`

设计要求：

- 回调标题牌匾外框、内框、左右翼片比例。
- 调整标题、副标题的字体大小、字重、字距、纵向间距与基线。
- 调整路径标题行、学期胶囊、三段路径卡的尺寸、内边距、行高与对齐方式。
- 确保路径卡中的图标位、文字块、箭头节奏与参考图一致。

关键约束：

- 标题框不允许再出现过宽、过薄、内边距错误的问题。
- 路径卡文字不允许再出现垂直不居中、行高偏差和边距不均问题。

## 文件改动范围

### 1. `app.js`

本轮职责：

- 增加正式导出背景内嵌逻辑。
- 修复 PNG 下载链路，确保使用内嵌背景的 SVG。
- 让 SVG 导出中的图片区、标题区、路径区继续向统一参数源收敛。
- 如必要，为视觉回归测试暴露适度可测的辅助函数。

### 2. `template-config.example.js`

本轮职责：

- 回调 `imageLayouts` 使上传后图片位置严格贴近 `模拟素材.jpg`。
- 回调 `textBlocks` 使标题、副标题、三阶段文案位置更准确。
- 回调 `layoutReference` 使标题牌匾、路径区、chip、footer 的几何更贴近参考图。

### 3. `styles.css`

本轮职责：

- 继续收敛预览层标题牌匾、路径卡、图片区、chip 的样式细节。
- 确保预览中的字体、边距、尺寸与正式导出视觉一致。
- 消除当前上传后布局在文字框样式上的偏差。

### 4. `app.test.js`

本轮职责：

- 增加背景内嵌相关单元测试。
- 增加导出 SVG 几何与布局参数的回归测试。
- 如适合，增加 PNG 下载辅助链路的可测行为测试。

## 测试方案

### 1. 单元测试

单元测试至少覆盖以下内容：

- 正式导出的 SVG 包含内嵌背景资源，而不是只保留外部相对路径。
- 内嵌背景逻辑不会影响预览背景路径读取。
- `imageLayouts` 回调后的关键坐标会进入 SVG。
- `titleFrame`、`pathArea`、`textBlocks` 的关键几何值会进入 SVG。
- 分类上传与标签条件隐藏逻辑不回退。

### 2. 视觉回归测试

视觉回归至少覆盖以下场景：

- 空状态预览。
- 只上传成绩单 / 证书。
- 只上传 Offer。
- 同时上传两类素材。

重点检查：

- 背景图色彩是否正常。
- 主图和辅图位置是否与参考结构一致。
- 标题框、路径区、标签和 footer 是否对齐。
- 文字框尺寸、边距、字重、行高、对齐方式是否符合参考效果。

### 3. 导出验证

导出验证至少包括：

- 生成 SVG 后，脱离原页面上下文独立查看，背景仍完整显示。
- 基于导出 SVG 生成 PNG，背景色彩与预览一致。
- PNG 不再出现灰底替代色。

## 验收标准

满足以下条件视为本轮完成：

- 预览、SVG、PNG 三者的背景显示效果一致，且不再出现灰底异常。
- 正式 SVG 不再依赖脆弱的外部相对路径背景引用。
- 上传后图片位置与 `模拟素材.jpg` 中的标准位置一致。
- 标题框、路径区、chip 与 footer 的文字框细节符合参考图节奏。
- 分类上传、标签条件隐藏、批量逻辑不回退。

## 风险与注意事项

- 若仅修 PNG 栅格化而不修 SVG 背景资源模型，灰底问题仍会在独立打开 SVG 时继续存在。
- 若仅微调 CSS 而不回调 `template-config.example.js`，正式导出仍会与预览产生偏差。
- 内嵌背景资源时需避免把预览链路也改成同步内嵌，防止引入不必要的页面开销。
