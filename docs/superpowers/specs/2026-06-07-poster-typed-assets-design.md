# 海报工具素材分类上传与固定背景约束设计

## 背景

当前项目是一个本地静态海报生成工具，核心文件为 `index.html`、`styles.css`、`app.js` 与 `app.test.js`。工具当前已经具备以下能力：

- 使用内置 SVG 引擎直接生成正式海报。
- 通过 `template-config.example.js` 指定正式使用的固定背景图。
- 支持单个录入和 CSV 批量导入。
- 支持 1 到 3 张图片的自动布局切换。

但当前实现仍存在两个关键问题：

1. 用户上传图片时没有素材类型概念，系统只按上传顺序把图片塞进统一的 `images[]` 数组中。
2. 预览与导出逻辑无条件渲染右侧标签，无法根据“成绩单/证书”与 “offer” 的真实上传状态做条件控制。

此外，`assets/1-poster.png` 作为测试图片暴露出明显偏差：该图片没有采用规定的固定品牌背景，而是使用了独立灰底成图，和 `assets/模拟素材.jpg` 所代表的正式标准不一致。

## 目标

本次设计目标如下：

- 梳理项目当前整体架构与素材管理链路，明确正式背景与用户上传素材的职责边界。
- 建立素材分类上传机制，支持“成绩单/证书”和“offer”两类素材独立上传，可单独上传任一类，也可同时上传两类。
- 在前端预览与正式导出中引入严格的条件渲染逻辑：若某一类素材未上传，则绝不渲染其对应标签。
- 保持预览与正式 SVG 输出的一致性，避免前端预览与导出结果出现分叉。
- 在单个录入与批量导入两条流程中统一落实上述规则。

## 非目标

本轮不做以下扩展：

- 不改变整体海报视觉风格与既有品牌规范。
- 不引入第三方模板服务或远程渲染平台。
- 不把用户上传的成品海报图当作正式背景使用。
- 不重新设计 CSV 文本字段结构，仅在必要处补充素材类型匹配规则。

## 当前架构梳理

### 1. 页面结构

`index.html` 当前分为四个主要区域：

- 单个学员录入区：输入标题、副标题、三阶段文案与图片。
- 海报预览区：模拟正式海报结构。
- 批量导入区：导入 CSV 与图片池。
- 结果区：展示导出的 SVG / PNG 结果。

### 2. 样式职责

`styles.css` 负责两类视觉：

- 工具后台界面本身的表单、列表和结果区样式。
- 海报预览卡片的固定布局样式，包括标题牌匾、路径框、图像标签、图像槽位和底部装饰。

### 3. 核心数据与渲染逻辑

`app.js` 当前承担以下职责：

- `mergeConfig()`：合并默认配置与外部配置。
- `createNormalizedRecord()` / `normalizeImages()`：清洗单条记录与图片数组。
- `validatePosterRecord()`：验证文本字段与图片数量。
- `resolvePosterImageLayout()` / `mapImagesToPosterSlots()`：根据 1 到 3 张图选择布局并映射图槽。
- `createPosterSvgMarkup()`：拼装正式 SVG 输出。
- `renderPreview()`：驱动页面预览。
- `handleManualImageUpload()` / `handleBatchImageUpload()` / `handleCsvUpload()`：处理上传行为。

### 4. 当前素材管理流程

现有素材可分为两类：

- 固定品牌素材：来自 `template-config.example.js` 中的 `assets.backgroundImage`，正式要求指向 `assets/香港灣區教育諮詢促進會 HONG KONG GREATER BAY AREA EDUCATIONAL PROMOTION ASSOCIATION.png`。
- 用户上传素材：当前统一作为 `images[]` 进入预览与导出，没有类型区分。

当前正式链路中，背景属于固定品牌层，用户只应上传右侧展示内容素材。两者职责不同，不能混用。

## 素材核查与问题分析

### 1. `assets/1-poster.png` 的现状

经核查，`assets/1-poster.png` 是一张已经排版完成的测试成图，具有以下特征：

- 使用灰色纯底背景。
- 顶部不含正式固定背景中的 Logo、机构名、夜景、金色底部装饰等品牌层。
- 右侧只有一张主图，缺少标准模拟素材中的多图层级结构。
- 虽然保留了口号、标题区、路径区和标签的大致位置，但其背景与成图角色已偏离系统标准。

### 2. 与 `assets/模拟素材.jpg` 的核心差异

`assets/模拟素材.jpg` 代表本项目要求落地的“模拟素材”标准，关键特征包括：

- 固定品牌背景已作为底层，不允许被测试图替代。
- 顶部包含机构 Logo 与机构名称。
- 右侧图片区具备“成绩单/证书”与 “offer” 两组视觉分区。
- 下半部分有夜景、金色上升箭头与底部金色结构装饰。

相比之下，`1-poster.png` 的差异体现在：

- 背景层错误：未采用规定的固定背景。
- 品牌层缺失：没有品牌底图中的 Logo 和完整装饰。
- 图片层级不完整：没有体现模拟素材中的双分区内容结构。
- 成图角色错误：它更像一个独立成品，而不是待套用到固定模板中的素材集合。

### 3. 结论

`1-poster.png` 可以作为“错误示例”或回归测试对照图使用，但不能作为正式海报背景，也不能替代 `template-config.example.js` 中指定的固定背景图。

## 设计方案

### 1. 数据模型调整

把当前“无类型统一图片数组”的模型升级为“按素材类型分组，再派生渲染结果”的模型。

单条记录在逻辑上拆分为：

- `certificateImages[]`：成绩单 / 证书类素材。
- `offerImages[]`：offer 类素材。
- `images[]`：按既定优先顺序由前两类派生出的最终布局输入数组。

每张图片对象新增统一元信息：

- `id`
- `file`
- `name`
- `previewUrl`
- `assetType`，取值为 `certificate` 或 `offer`

批量图片池也采用相同结构，避免单个流程与批量流程出现两套模型。

### 2. 图片合成顺序

为保证现有 1 到 3 图布局尽量少改，正式渲染数组按如下顺序派生：

- 先放入 `certificateImages`
- 再补入 `offerImages`
- 最终截取前 3 张进入版式布局

这样做的原因：

- “成绩单 / 证书”通常对应右侧上方主展示区域。
- “offer” 往往作为补充素材进入下方辅图位。
- 保持和当前既有版式的认知一致，避免本轮同时重做全部坐标系统。

若仅上传 offer，则允许 offer 直接占用第一张可用槽位；但“成绩单 / 证书”标签必须隐藏，“真实Offer”标签可以显示。

### 3. 单个录入上传模块重构

单个录入区改为显式分类上传：

- 增加“上传成绩单 / 证书”文件输入。
- 增加“上传 Offer”文件输入。
- 两类素材分别展示已上传列表。
- 删除逻辑按所属分类独立处理。

单个录入状态中不再只维护一个扁平 `manualRecord.images`，而是维护：

- `manualRecord.certificateImages`
- `manualRecord.offerImages`

渲染前再通过派生函数生成最终 `manualRecord.images`。

### 4. 批量导入流程重构

批量流程也采用类型化上传，但尽量不推翻当前 CSV 结构。

设计如下：

- CSV 仍负责导入文本字段。
- 批量图片区改为两个上传入口：
  - 批量导入成绩单 / 证书图片
  - 批量导入 offer 图片
- 每个上传池中的图片都带上 `assetType`。
- 匹配记录时，把两类池合并进统一匹配逻辑，但保留类型信息。

记录匹配完成后，每条批量记录中将得到：

- `certificateImages`
- `offerImages`
- 派生后的 `images`
- `missingImages`

如果 CSV 当前仍只提供 `image1~image3` 这类文件名字段，则匹配逻辑默认按“文件名命中图片池”方式归类；类别来源以图片上传入口为准，而不是从 CSV 文本字段推导。

### 5. 条件渲染规则

这是本轮最核心的前端规则。

#### 预览区

- 如果 `certificateImages.length === 0`，则隐藏“真实成绩单 / 证书”标签。
- 如果 `offerImages.length === 0`，则隐藏“真实Offer”标签。
- 如果某类没有图片，则不为该类保留空白标签占位。
- 图片槽位只显示实际映射到的图片，不为未上传的类别强行保留专属空框。

#### SVG 导出

- 正式 SVG 中采用与预览相同的标签显隐规则。
- `createPosterSvgMarkup()` 只有在对应类型存在时才输出对应 chip 的 SVG markup。
- 不允许出现“没有 offer 图但导出中仍保留真实Offer 标签”的情况。

### 6. 固定背景约束

正式背景的约束进一步明确为：

- 预览背景与正式导出背景都只能来自配置中的固定背景。
- `assets/1-poster.png` 不允许进入背景链路。
- 若配置背景缺失，页面只能报错提醒，不允许回退到用户上传素材或测试图。

这条规则既是视觉规范要求，也是产品职责边界：用户上传的是案例素材，而不是模板背景。

## 关键实现边界

### 1. 新增或调整的核心函数

建议新增以下逻辑单元：

- `normalizeTypedImages(images, assetType)`
- `buildDerivedRecordImages(record)`
- `getRecordAssetVisibility(record)`
- `matchRecordImagesByType(records, typedImagePool)`

同时调整以下已有函数：

- `createNormalizedRecord()`
- `validatePosterRecord()`
- `mapImagesToPosterSlots()`
- `createPosterSvgMarkup()`
- `renderManualImageList()`
- `renderPreview()`
- `handleManualImageUpload()`
- `handleBatchImageUpload()`
- `matchRecordImages()`

### 2. 验证规则

验证规则更新为：

- 标题、副标题、三阶段文案仍为必填。
- 图片验证从“统一上传 1 到 3 张”改为“分类上传后合计 1 到 3 张”。
- 若两类都未上传，则报错。
- 若分类上传后总数超过 3 张，则报错。

### 3. UI 约束

页面文案需要同步更新，避免误导用户继续按“无类型图片池”理解系统：

- 单个上传提示文案改为说明两类素材可任选其一或同时上传。
- 批量上传提示文案改为说明应按素材类别分入口上传。
- 生成前摘要中增加两类素材数量统计。

## 测试策略

### 1. 自动化测试

`app.test.js` 需要补充或重构以下测试：

- 仅上传成绩单图片时：
  - 记录验证通过。
  - 预期仅显示成绩单标签。
  - SVG 中不包含 `真实Offer` 标签。
- 仅上传 offer 图片时：
  - 记录验证通过。
  - SVG 中不包含 `真实成绩单 / 证书` 标签。
  - offer 标签可正常出现。
- 两类都上传时：
  - 两个标签都出现。
  - 图片派生顺序符合设计。
- 总图数为 0 或超过 3 时：
  - 验证失败。
- 背景缺失时：
  - 仍报固定背景缺失错误。
- 固定背景存在时：
  - 导出使用配置中的背景，而不是测试图片。

### 2. 手工验证

手工测试要覆盖以下路径：

- 单个录入只传成绩单。
- 单个录入只传 offer。
- 单个录入两类都传。
- 批量 CSV + 分类图片池匹配。
- 删除单类图片后标签即时隐藏。
- 固定背景可正常显示，且不会被 `1-poster.png` 替换。

## 验收标准

达到以下条件即可视为本次需求完成：

- 已完成项目架构与素材管理流程梳理，并明确固定背景与用户素材的边界。
- 已确认 `assets/1-poster.png` 不符合“模拟素材”标准，且未采用规定固定背景。
- 单个录入与批量导入均支持“成绩单 / 证书”与 “offer” 分类上传。
- 任一类别未上传时，其对应标签不会出现在预览或导出中。
- 正式预览与正式 SVG 输出继续强制使用固定背景。
- 自动化测试与手工测试均验证通过。

## 风险与注意事项

- 当前布局槽位并非为“类型感知”而设计，若后续希望让不同类型始终固定占据指定区域，可能需要在下一轮进一步把布局模型从“按顺序放图”升级为“按类型映射槽位”。
- 批量 CSV 目前没有显式的类型字段，因此图片类别只能依赖上传入口判定，后续如需更强可追溯性，可考虑为 CSV 增加类型列。
- 如果页面上仍保留旧的“统一上传图片”描述，容易让用户误解本轮设计，应一并清理。
