# 海报工具收尾优化与分期验收设计

## 背景

当前项目已经具备以下基础能力：

- 固定背景海报预览与正式导出链路已经建立。
- 正式 SVG 导出已支持背景资源内嵌，灰底异常问题已被修复。
- 素材分类上传、标签条件显隐、批量匹配与正式导出流程已经可用。
- `template-config.example.js` 已开始承担预览与导出共享几何参数的职责。

但项目仍处于收尾阶段，尚存在以下问题：

1. `styles.css` 中仍有按阶段叠加的样式定义，存在冗余、分散和局部冲突风险。
2. 预览层的图片区仍保留部分百分比硬编码逻辑，与配置驱动的正式导出尚未完全统一。
3. 当前缺少系统化的视觉回归流程，无法稳定发现后续 CSS 与布局迭代带来的回退。
4. 用户要求的跨浏览器、弱网、暗黑模式、多分辨率等最终验收维度，无法在当前仓库内一次性全部自动化落地，需要分阶段承接。

## 目标

本轮设计的目标是：

- 将项目收尾工作拆分为可落地的两期方案，避免把“代码实现”与“人工终验”混在一起。
- 在第一期内完成项目内部可直接实现的 CSS 收敛、图片布局优化、图片加载策略统一与视觉回归基线建设。
- 让第一期的主验收对象聚焦于“海报预览与正式 SVG/PNG 导出的一致性”。
- 在第二期中交付可执行的人工验收矩阵、记录模板与问题闭环机制，用于跨浏览器、弱网、暗黑模式和真机终验。

## 非目标

本轮明确不做以下内容：

- 不在第一期中宣称已自动完成 Safari、Edge、真机和所有弱网场景的真实验收。
- 不引入新的前端框架、组件库或模板引擎。
- 不重新设计新的海报模板结构或新增多模板支持。
- 不脱离当前静态 HTML、CSS、JavaScript 架构做大规模重构。
- 不将第二期的人工验收报告伪装为第一期自动化测试结果。

## 分期方案

### 第一期：仓库内可直接落地的收尾优化

第一期仅包含当前项目内部可以真实编码、执行与验证的内容：

- `styles.css` 样式体系收敛与规范化
- 预览与导出的图片布局规则统一
- 图片加载、占位、裁剪、防变形与懒加载策略统一
- “预览 + 正式 SVG/PNG” 为核心对象的视觉回归
- 桌面端与移动端主要断点的浏览器内验收

第一期必须输出真实代码改动、可重复执行的测试结果和可回归的场景资产。

### 第二期：人工终验与发布前闭环

第二期不在本轮直接实现为自动化能力，而是输出可执行的人工终验包，覆盖：

- 浏览器：Chrome、Firefox、Safari、Edge 最新稳定版
- 分辨率：1920、1440、1080
- 移动端：iPhone 主流宽度与安卓主流宽度
- 环境：弱网、暗黑模式、必要时的系统缩放或高 DPI 差异
- 业务链路：上传、预览、导出、批量处理、结果查看

第二期必须输出检查清单、记录模板、问题优先级字段与闭环状态字段。

## 核心方案

### 1. CSS 收敛方案

涉及文件：

- `styles.css`

设计要求：

- 按职责重组样式结构，至少分为：
  - 全局 token 与语义变量
  - 页面壳层与布局容器
  - 表单与通用组件
  - 海报预览专属组件
  - 图片、标签与结果区组件
- 收敛重复的色值、阴影、边框、圆角、字号与间距定义。
- 保留当前品牌视觉方向，不重新换风格，只提升一致性与可维护性。
- 对重复选择器与覆盖链进行清理，避免“后加补丁覆盖前面定义”的继续累积。

关键约束：

- 不允许因为收敛样式而破坏现有海报品牌视觉。
- 不允许把工具页壳层样式和海报预览样式继续耦合得更深。

### 2. 图片布局与几何统一方案

涉及文件：

- `template-config.example.js`
- `app.js`
- `styles.css`

设计要求：

- 预览与正式导出必须收敛到同一套图片布局规则。
- `1-image`、`2-image`、`3-image` 三套配置继续以 `template-config.example.js` 为唯一几何来源。
- 预览层不再长期依赖独立的百分比硬编码图片位作为最终真值。
- 图片标签 `真实成绩单 / 证书` 与 `真实Offer` 的位置需继续与配置和导出结果保持一致。

关键约束：

- 某一类素材未上传时，对应标签仍必须隐藏。
- 图片不得拉伸、挤压、越界或遮挡标题牌匾与路径区。

### 3. 图片加载与占位优化方案

涉及文件：

- `index.html`
- `app.js`
- `styles.css`

设计要求：

- 统一图片容器的比例和裁剪策略，明确：
  - 预览图片区使用何种 `object-fit`
  - 结果区缩略图使用何种 `object-fit`
  - 导出区如何保持与预览裁剪一致
- 增加统一的加载占位态与失败兜底态，避免空白、抖动和尺寸塌陷。
- 对非关键区域图片启用统一懒加载策略；关键海报预览主视觉不做激进懒加载，优先保证稳定展示。
- 统一图片加载性能策略，减少不必要的重复解码或无意义重排。

关键约束：

- 不允许为了追求懒加载而让核心海报预览延迟展示或造成视觉闪烁。
- 不允许出现不同区域对同类图片采用不同裁剪规则导致观感不一致。

### 4. 视觉回归方案

涉及文件：

- 新增视觉回归脚本与结果目录
- `app.test.js` 继续承担单元测试职责

设计要求：

- 单元测试与视觉回归分层，不把截图流程塞入 `node:test`。
- 视觉回归第一期的主验收对象是：
  - 海报预览
  - 正式 SVG/PNG 导出结果
- 工具页表单、批量导入区与结果区只做基础回归，不作为第一期最高优先级视觉比对对象。

第一期视觉回归至少覆盖以下场景：

- 空状态
- 仅上传成绩单 / 证书
- 仅上传 Offer
- 同时上传两类素材
- 1 张图
- 2 张图
- 3 张图
- 桌面主断点
- 移动主断点

第一期重点检查：

- 预览与导出背景是否一致
- 图片位置、大小、旋转与裁剪是否一致
- 标题框、路径区、标签、footer 是否回退
- 图片占位、加载后显示和隐藏标签逻辑是否稳定

### 5. 第二期人工验收方案

涉及文件：

- 新增验收清单文档
- 新增问题记录与闭环模板

设计要求：

- 第二期不宣称自动执行所有目标环境，而是输出可执行、可记录、可跟进的问题闭环包。
- 验收矩阵至少包含：
  - 平台 / 浏览器
  - 分辨率 / 设备宽度
  - 网络条件
  - 主题模式
  - 关键业务流程
  - 预期结果
  - 实际结果
  - 问题级别
  - 修复状态

关键约束：

- 每个检查项必须可直接执行，不能写成空泛描述。
- 必须能支撑最终上线前人工走查和回归复验。

## 模块拆分

### 模块一：样式治理模块

职责：

- 统一全局视觉 token
- 重组样式结构
- 清理冗余和冲突 CSS

主要文件：

- `styles.css`

### 模块二：海报几何与图片规则模块

职责：

- 统一预览与导出的图片布局规则
- 把图片几何和标签位置继续收口到配置

主要文件：

- `template-config.example.js`
- `app.js`
- `styles.css`

### 模块三：图片加载与占位模块

职责：

- 统一图片加载、占位、失败兜底和懒加载策略

主要文件：

- `index.html`
- `app.js`
- `styles.css`

### 模块四：视觉回归与终验模块

职责：

- 搭建视觉回归脚本
- 输出第一期自动化回归结果
- 输出第二期人工验收包

主要文件：

- 视觉回归脚本文件
- 验收文档
- `app.test.js` 的补充测试

## 测试策略

### 1. 单元测试

单元测试继续覆盖：

- 导出与预览共享的关键几何参数
- 标签条件显隐逻辑
- 图片布局规则入口
- 导出链路的稳定性

### 2. 视觉回归测试

视觉回归第一期聚焦：

- 预览截图
- 正式导出结果截图
- 关键断点下的主要布局稳定性

视觉回归结果需要可重复执行，并具备明确场景命名。

### 3. 人工终验

第二期人工终验覆盖：

- 分辨率
- 浏览器
- 移动端主流宽度
- 弱网
- 暗黑模式
- 关键业务流程

终验必须使用结构化清单和问题记录模板。

## 验收标准

第一期完成标准：

- `styles.css` 中重复、冲突和分散的样式定义显著收敛。
- 预览与正式导出的图片布局规则进一步统一，不再长期依赖双轨几何。
- 图片在关键场景下不拉伸、不溢出、不异常占位。
- 存在可重复执行的视觉回归流程，且覆盖核心海报场景。
- 预览与导出的主视觉结果在关键场景中保持一致。

第二期交付标准：

- 存在完整的人工验收矩阵。
- 存在问题记录与闭环模板。
- 浏览器、分辨率、网络、主题与业务流程检查项完整可执行。

## 风险与注意事项

- 如果只做 CSS 清理而不统一图片几何来源，预览与导出仍可能继续分叉。
- 如果只做截图而不建立场景命名和基准管理，视觉回归很快会失去可维护性。
- 如果把海报主视觉区也做成激进懒加载，容易引入首屏闪烁和验收误判。
- 如果第一期强行承诺全浏览器真机自动化，会制造无法兑现的验收范围，应通过第二期人工终验包承接。
